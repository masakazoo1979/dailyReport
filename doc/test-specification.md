# 営業日報システム テスト仕様書

## 1. テスト概要

### 1.1 目的

営業日報システムの品質を保証するため、機能要件、非機能要件を満たすことを検証する。

### 1.2 対象システム

- システム名: 営業日報システム
- バージョン: 1.0

### 1.3 テスト実施体制

| 役割               | 担当者 | 責任                   |
| ------------------ | ------ | ---------------------- |
| テストマネージャー | -      | テスト計画、進捗管理   |
| テストリーダー     | -      | テスト設計、レビュー   |
| テスター           | -      | テスト実施、不具合報告 |

---

## 2. テスト戦略

### 2.1 テストレベル

| テストレベル   | 目的                     | 実施者   | 実施時期             |
| -------------- | ------------------------ | -------- | -------------------- |
| 単体テスト     | 個別モジュールの動作検証 | 開発者   | 開発中               |
| 結合テスト     | モジュール間の連携検証   | 開発者   | 開発完了後           |
| システムテスト | システム全体の機能検証   | テスター | 結合テスト完了後     |
| 受け入れテスト | ユーザー要件の充足確認   | ユーザー | システムテスト完了後 |

### 2.2 テストタイプ

| テストタイプ         | 目的                   | 対象             |
| -------------------- | ---------------------- | ---------------- |
| 機能テスト           | 機能要件の検証         | 全機能           |
| UI/UXテスト          | 画面表示・操作性の検証 | 全画面           |
| APIテスト            | API仕様の検証          | 全エンドポイント |
| 権限テスト           | アクセス制御の検証     | 役割別機能       |
| バリデーションテスト | 入力検証の確認         | 入力項目         |
| セキュリティテスト   | 脆弱性の検証           | 認証、入力項目   |
| パフォーマンステスト | 性能要件の検証         | 主要機能         |
| 互換性テスト         | ブラウザ・デバイス対応 | 全画面           |

---

## 3. テスト範囲

### 3.1 テスト対象

#### 機能範囲

- ✅ ログイン・ログアウト機能
- ✅ ダッシュボード機能
- ✅ 日報管理機能（登録/編集/削除/提出/承認/差し戻し）
- ✅ 訪問記録管理機能
- ✅ コメント機能
- ✅ 顧客マスタ管理機能
- ✅ 営業マスタ管理機能
- ✅ 権限制御
- ✅ バリデーション
- ✅ エラーハンドリング

#### 非機能範囲

- ✅ パフォーマンス（レスポンス時間）
- ✅ セキュリティ（認証、認可、脆弱性）
- ✅ ブラウザ互換性
- ✅ レスポンシブデザイン

### 3.2 テスト対象外

- ❌ インフラストラクチャの冗長性
- ❌ 障害復旧
- ❌ 負荷テスト（大規模同時アクセス）
- ❌ 外部システム連携（本バージョンでは未実装）

## 8. 不具合管理

### 8.1 不具合重要度

| 重要度 | 説明                   | 対応               |
| ------ | ---------------------- | ------------------ |
| 致命的 | システムが動作しない   | 即座に修正         |

## 9. テスト実施計画

### 9.1 スケジュール

| フェーズ       | 期間 | 担当           | 成果物                   |
| -------------- | ---- | -------------- | ------------------------ |
| テスト設計     | 5日  | テストリーダー | テスト仕様書             |
| テスト環境構築 | 3日  | インフラ担当   | テスト環境               |
| 単体テスト     | 10日 | 開発者         | テスト結果               |
| 結合テスト     | 7日  | 開発者         | テスト結果               |
| システムテスト | 10日 | テスター       | テスト結果、不具合リスト |
| 不具合修正     | 5日  | 開発者         | 修正済みシステム         |
| 回帰テスト     | 3日  | テスター       | テスト結果               |
| 受け入れテスト | 5日  | ユーザー       | 受け入れ完了             |

### 9.2 テスト完了基準

以下の条件をすべて満たした場合、テスト完了とする：

1. すべてのテストケースが実施済み
2. 致命的・高重要度の不具合が0件
3. 中重要度の不具合が10件以下
4. すべての不具合が修正完了またはクローズ
5. 回帰テストで新たな不具合が発見されていない
6. パフォーマンス要件を満たしている
7. セキュリティテストに合格している
8. ユーザー受け入れテストに合格している
